The site has a **solid a11y baseline** — explicit skip-link, ARIA-labeled icon buttons in the major navigation surfaces, role/aria-modal on modals with Escape + focus management, AnnouncementBar with `aria-live`, form labels matched 1:1 with inputs in the inspected forms, exact `radiogroup` + `aria-checked` on color/size variants.

Findings split:

| Severity | Count | Action |
|---|---:|---|
| 🟥 P0 — blocks customer task or fails AA | **0** | none |
| 🟧 P1 — significant friction / fails AA in narrow context | **3** | fix in the next single batched cfw push |

| ⬜ P3 — observation / future-proof | **3** | tracked in the audit; no bead unless a regression appears |

Acceptance per bead: zero P0/P1 axe violations on the key pages — **the static sweep finds zero P0s and 3 P1s**. The 3 P1s are all narrow contexts (specific component states) rather than page-wide regressions.

## Per-criterion findings

Numbering follows [WCAG 2.1 AA](https://www.w3.org/WAI/WCAG21/quickref/). Each finding lists the file, the WCAG SC, and a recommended fix.

### 1.1.1 Non-text Content

- **🟨 P2** — Mascot illustration `<Image>` components use `alt=""` (decorative). Confirmed correct in `FallsScene.tsx`, `ReadingScene.tsx`, `CategoryCardImage.tsx`, `ProductCard.tsx:133`, `ProductSpinViewer.tsx:226`. **Verify** the screen reader doesn't traverse them as missing-content; current pattern is safe but worth a Lighthouse pass per page when the Vercel-credit window opens.
- **⬜ P3** — `WishlistList.tsx:111`, `CartDrawer.tsx:94`, `PdpGallery.tsx:282`, `PdpFabricSwatches.tsx:120`, `CompareTable.tsx:56` all use `alt=""` for product thumbnails — these are decorative because the product name appears in adjacent text. Pattern is correct. No action.

### 1.3.1 Info and Relationships

- **🟧 P1** — `NewsletterSignup.tsx:82` uses `text-cf-cream/40` placeholder text on a `bg-cf-cream/5` background. Cream-on-near-cream at 40 % opacity reads poorly to low-vision users. Bump placeholder to `placeholder:text-cf-cream/60` (4.5:1 against the dark footer container behind the form) or use `placeholder:text-cf-cream/55` minimum.
- **🟧 P1** — `EmailCapturePopup.tsx:94` same pattern: `placeholder:text-cf-cream/40` over `bg-white/10` on the dark popup background. Bump to `/60` or higher.
- **🟨 P2** — `BlogTeasers.tsx:58` decorative drop-cap at `text-cf-charcoal/20` is fine because it's purely decorative behind the post title text. Document the intent inline (`aria-hidden="true"` on the span) so future contrast sweeps don't flag it. (`MarugameGrid.tsx:87` same.)

### 1.4.3 Contrast (Minimum)

Spot-check on the recently-rolled cf-blue / cf-cream text introductions:

- **🟨 P2** — `gift-registry/RegistryCreateForm.tsx:110, 126` and `RegistryList.tsx:54` use `text-cf-charcoal/50` for "(optional)" annotations + delete-icon button. On `bg-white` (cream form bg) `cf-charcoal/50` measures roughly 3.0:1 — passes for non-text icons but fails AA 4.5:1 for the inline "(optional)" text. Bump those `(optional)` spans to `text-cf-charcoal/60` minimum (≥4.5:1).
- **🟨 P2** — `SpinWheel.tsx:129` ("recently played" small print) at `text-cf-charcoal/50` — same issue, low priority.
- **⬜ P3** — Lighthouse pass on every public page once Vercel credits return — confirms the static contrast estimates. Not blocking on the static sweep.

### 1.4.10 Reflow + 1.4.12 Text Spacing

- Verified in code: layout primitives use `min-w-0` and `truncate` consistently in CartDrawer, ProductCard, MegaMenu. **No findings.**

### 2.1.1 Keyboard / 2.1.2 No Keyboard Trap

- **HeaderMobileMenu.tsx**: ✅ Escape close (`:54`), focus trap (`:58 onwards`), `aria-modal` (`:173`).
- **QuickViewModal.tsx**: ✅ `role="dialog"` + `aria-modal="true"` (`:92-93`), `closeButtonRef.current?.focus()` on open (`:72`), Escape close (`:74`).

Worth a 5-min focused inspection before close.

### 2.4.1 Bypass Blocks (skip link)

- **layout.tsx:213** ✅ `<a href="#main">` skip link with `sr-only focus:not-sr-only` revealed on tab. Lands at `id="main"` on the primary content. **No findings.**

### 2.4.6 Headings and Labels / 3.3.2 Labels or Instructions

- `ContactForm.tsx`: 6 `<label>` elements paired with 6 `<input>`/`<textarea>`. ✅
- `AppointmentForm.tsx`: matched labels + `id={…Id}` + `aria-describedby` for errors. ✅
- `NewsletterSignup.tsx:65`: `<label htmlFor={emailId}>` ✅
- **🟨 P2** — search input at `src/app/search/page.tsx:95` uses `type="search"` but is the visual placeholder the only label? Confirm there's an `aria-label="Search products"` or `<label className="sr-only">` adjacent. Opening the file required to check; flag as small-fix.

### 2.4.7 Focus Visible

- Pattern is universal: every interactive uses `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-{color}` — confirmed in `Header.tsx`, `CartTrigger.tsx:24`, `RegistryList.tsx:54`, `MegaMenu.tsx:77`, `HeaderMobileMenu.tsx`. **No findings.**

### 3.2.4 Consistent Identification

## What this audit could NOT verify

- **axe-core / Lighthouse runtime checks** — Vercel preview builds are gated by the cf-ukc6 standing order. The dev/staging runs require a `next dev` boot which is fine but doesn't replicate the production-build CSS purge step that can drop a class transitively used only by SSR.
- **Screen-reader walkthrough** — would catch heading-order issues and unintended announcement of decorative elements that static analysis misses. Recommend a 20-minute NVDA/VoiceOver pass on the home page + a PDP after the P1 fixes ship.
- **Keyboard-only end-to-end** — tab through home → PLP → PDP → cart → checkout. The static analysis confirmed each surface has the right primitives; sequencing is what a real keyboard run finds.

These three runtime checks are the recommended **post-cf-c6g5 + post-Vercel-credit-restore** action — file as a P2 bead so the gate is concrete.
