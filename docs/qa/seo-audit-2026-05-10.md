## What's already in place

### Sitewide (layout.tsx)
- `metadataBase` resolves relative URLs (canonical/alternates that come from per-page exports).
- Default `title`, `description`, `openGraph: { siteName, type, locale, images: [DEFAULT_OG_IMAGE] }`, `twitter: { card: "summary_large_image" }`.
- `verification: resolveVerification()` for Search Console / Bing.
- `<JsonLd id="jsonld-org" schema={buildOrganizationSchema(siteUrl)} />` mounted globally — every page emits Organization structured data.

### PDP — `src/app/products/[slug]/page.tsx`
- `generateMetadata` with title/description/openGraph (site-aware).
- `JsonLd id="jsonld-product"` — `Product` schema with `name, description, image, url, offers { price, priceCurrency: USD, availability, url }`. Optional `aggregateRating` + `review` arrays gated on real review data (cf-xe54 — no fabricated stats).
- `JsonLd id="jsonld-breadcrumb"` — Home → Shop → Product.
- Computed `canonicalUrl = ${siteUrl}/products/${slug}` is fed to the schema **but not declared as `alternates: { canonical }`** on the metadata export. (See §P1.1.)

### PLP — `src/app/shop/[category]/page.tsx`
- `generateMetadata` builds per-category title/description.
- searchParams (`sort`, `page`, `priceMin`, `priceMax`, `inStock`) drive URL variants.
- **No canonical declared** — every filtered URL is a separate canonical from Google's POV. (See §P1.2.)

### Other schemas
- `/blog/[slug]` — `BlogPosting` schema + per-post `openGraph`. ✅
- `/faq` — `FAQPage` schema. ✅
- `/visit` — `LocalBusiness` schema. ✅
- `/sustainability`, `/near/[city-slug]`, `/reviews` — bespoke JSON-LD per file. ✅

## Findings

### 🟧 P1.1 — PDP missing explicit `alternates: { canonical }`

**File**: `src/app/products/[slug]/page.tsx`

The PDP's `generateMetadata` builds title + description + openGraph but does not return an `alternates: { canonical }` field. Today the canonical-URL signal lives **only** in the Product JSON-LD (`url: canonicalUrl`). Search engines do honour the JSON-LD URL, but the metadata-level `<link rel="canonical">` is the authoritative HTML signal — without it a PDP visited via a tracking parameter (`?utm_source=…`, `?ref=…`) is canonicalized to itself by default, splitting link equity.

**Fix** (1-line addition to the metadata return):
```ts
alternates: { canonical: `${siteUrl}/products/${slug}` },
```

### 🟧 P1.2 — PLP missing canonical

**File**: `src/app/shop/[category]/page.tsx`

The PLP supports `?sort=…&page=N&priceMin=…&priceMax=…&inStock=1`, producing many duplicate-content URLs for the same category. Each variant is its own canonical from Google's POV today.

**Fix**: in `generateMetadata`, return:
```ts
alternates: { canonical: `${siteUrl}/shop/${categorySlug}` },
```

This points every filtered URL back to the base category. (Pagination is a debate — `page=2` is genuinely different content; the right canonical for paginated PLPs is the same `/shop/${category}` page-1 URL, with the paginated version's content treated as supplementary. Stick with the base canonical for v1; revisit after we know whether crawl budget is being wasted.)

Both P1 fixes belong in **one small PR**. ~6 lines total.

### 🟨 P2 — Twitter card uses default `summary_large_image` everywhere

`twitter` block in layout sets only `card: "summary_large_image"`. Pages don't override `twitter.title` or `twitter.description`. For non-OG-aware Twitter readers this means the card description is whatever Twitter pulls from the page's first paragraph (often a navigation breadcrumb).

**Action**: small enhancement — add per-page `twitter: { title, description, images }` when overriding `openGraph` (cleanest pattern: hoist into a helper). Low priority.

File as `cf-?? cfw SEO: per-page twitter card customization`.

### 🟨 P2 — No `Article` schema on `/our-story`, no `BreadcrumbList` outside PDPs

PLPs and content pages would benefit from `BreadcrumbList` (better crumb display in SERPs). And `/our-story` could carry an `AboutPage` or `Article` schema.

File as `cf-?? cfw SEO: BreadcrumbList on PLP + AboutPage on /our-story`.

### ⬜ P3 — sitemap doesn't include `/near/[city-slug]` pages

Spot-check: `sitemap.ts` enumerates `STATIC_PATHS`, `SHOP_CATEGORIES`, `listAllProducts()`, `listAllPostSlugs()`. The `/near/[city-slug]` SEO landing pages (cf-l6aj.21 added them to a separate `near-cities-sitemap.xml`) — confirm they're cross-referenced from the main sitemap or that Search Console knows about both.

### ⬜ P3 — Image alt text quality

Out of scope for metadata audit but adjacent: PLP `<Image alt={product.name}>` is correct; PDP gallery uses `alt=""` for additional images (decorative since the main image carries the name). Consistent with the cf-7tkf a11y audit's findings.

## What this audit could NOT verify

Per cf-ukc6, no preview build, so the following are deferred:
- **Google Rich Results Test** against live URLs — the JSON-LD shapes look right but the Test catches type-narrowing issues (e.g. `priceCurrency` accepted formats) that static analysis misses.
- **Search Console coverage report** — confirms what Google actually indexes vs the sitemap.
- **Lighthouse SEO score** per page — would catch missing `<title>` collisions and other runtime issues.
- **Canonical conflict check** — if a page has both a metadata-level canonical AND a `<link rel="canonical">` from another source, Google picks one inconsistently. Static scan didn't surface conflicts but a runtime DOM check would.

These four are the recommended **post-cf-c6g5 + post-credit-restore** action — file as a P2 bead so the gate is concrete.
